# Booking Automation

Система автоматизации записи для малого бизнеса:
**форма на сайте → событие в Google Календаре владельца → WhatsApp-подтверждение клиенту**.

Стек: Next.js 15 (App Router, Route Handlers), TypeScript (strict, без `any`),
Tailwind CSS 4, Zod (серверная валидация). Готово к деплою на Vercel.

## Архитектура (feature-based)

```
booking-automation/
├── .env.example                  # Карта окружения (см. ниже)
├── src/
│   ├── app/                      # Тонкий слой маршрутизации Next.js
│   │   ├── layout.tsx
│   │   ├── page.tsx              # Страница с формой записи
│   │   └── api/
│   │       ├── booking/route.ts       # POST — создание брони (ядро системы)
│   │       └── availability/route.ts  # GET — свободные слоты дня
│   ├── features/
│   │   └── booking/              # Вся доменная логика фичи «запись»
│   │       ├── schema.ts         # Zod-схемы запросов (источник истины)
│   │       ├── slot-rules.ts     # Бизнес-правила слота + генерация сетки
│   │       ├── types.ts          # Контракт API (общий для сервера и клиента)
│   │       └── components/
│   │           └── BookingForm.tsx    # Клиентская форма (Tailwind)
│   └── lib/                      # Инфраструктурные адаптеры (server-only)
│       ├── env.ts                # Zod-валидация process.env, кеш
│       ├── errors.ts             # BookingError: код → HTTP-статус
│       ├── google-calendar.ts    # JWT-auth, freebusy, insert, арбитраж гонок
│       ├── whatsapp.ts           # Интерфейс провайдера: Green-API / Twilio
│       ├── slot-mutex.ts         # Мьютекс по ключу слота (внутри инстанса)
│       ├── rate-limit.ts         # Fixed-window лимитер по IP
│       └── timezone.ts           # Intl-конверсия таймзон без зависимостей
└── package.json
```

Принципы:

- `app/` — только маршрутизация и оркестрация; доменная логика живёт в
  `features/booking`, интеграции — в `lib/`.
- Контракт ответа (`features/booking/types.ts`) импортируется и роутом,
  и формой — клиент и сервер не могут разъехаться по типам.
- Все секреты читаются только в `lib/env.ts` на сервере; ни одной переменной
  с префиксом `NEXT_PUBLIC_` в проекте нет.

## Настройка

1. `cp .env.example .env.local` и заполните значения (инструкции по получению
   ключей Google и WhatsApp — в комментариях внутри `.env.example`).
2. Дайте сервис-аккаунту право «Внесение изменений в мероприятия» на календарь
   владельца (настройки календаря → доступ для отдельных пользователей).
3. `npm install && npm run dev` → http://localhost:3000

Деплой на Vercel: импортируйте репозиторий, укажите Root Directory =
`booking-automation`, перенесите переменные из `.env.local` в
Project → Settings → Environment Variables. `GOOGLE_PRIVATE_KEY` вставляйте
как одну строку с `\n` — приложение развернёт переводы строк само.

## Безопасность и надёжность (P0)

**Валидация в три слоя.** Zod-схема (форма данных: E.164-телефон, реальная
календарная дата, длины полей) → бизнес-правила слота (будущее время, рабочие
часы, выравнивание по сетке, горизонт записи) → проверка занятости в календаре.
Клиентская валидация — только UX; сервер ничему из браузера не доверяет.

**Защита от двойного бронирования — три рубежа:**

1. *Мьютекс по ключу слота* (`slot-mutex.ts`) сериализует конкурентные запросы
   на одно время внутри одного инстанса (даблклик, повторная отправка).
2. *freebusy-проверка* перед созданием события отклоняет занятые слоты (409).
3. *Insert-then-verify* (`resolveInsertRaceWinner`): после `events.insert`
   перечисляются все события, пересекающие слот. Победитель определяется
   детерминированно (самое раннее `created`, tie-break по `id`), проигравший
   инстанс сам удаляет своё событие и возвращает клиенту 409. Так закрывается
   гонка между разными serverless-инстансами, где мьютекс в памяти бессилен.

**Прочее:**

- Rate limit 5 запросов/мин по IP (429 + `Retry-After`); honeypot-поле против
  ботов (бот получает фиктивный «успех», ничего не создавая).
- Ошибка отправки WhatsApp **не отменяет** бронь: событие уже в календаре,
  клиент получает `whatsappNotified: false`, инцидент пишется в лог.
- `/api/availability` отдаёт только флаги «свободно/занято» из freebusy —
  названия чужих событий наружу не утекают.
- Каждый ответ помечен `x-request-id` для трассировки в логах Vercel.

## Контракт API

`POST /api/booking` — тело: `{ name, phone, date, time, comment? }`.

| Статус | Код ошибки          | Когда                                  |
|--------|---------------------|----------------------------------------|
| 201    | —                   | Бронь создана (`data.whatsappNotified`)|
| 400    | `VALIDATION_ERROR`  | Битый JSON / нет параметра             |
| 409    | `SLOT_TAKEN`        | Слот занят или проигран в гонке        |
| 422    | `VALIDATION_ERROR`, `SLOT_INVALID` | Поля не прошли Zod / бизнес-правила |
| 429    | `RATE_LIMITED`      | Превышен лимит запросов                |
| 502    | `CALENDAR_ERROR`    | Google Calendar недоступен             |
| 503    | `CONFIG_ERROR`      | Окружение сконфигурировано неверно     |
| 500    | `INTERNAL_ERROR`    | Непредвиденный сбой                    |

`GET /api/availability?date=ГГГГ-ММ-ДД` → `{ success, data: { date, timeZone, slots: [{ time, available }] } }`
